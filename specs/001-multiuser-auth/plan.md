## Summary

Add two deployment modes (self-hosted and public) to the Vagrantfile Generator application. Self-hosted mode maintains current behavior with no authentication. Public mode implements multi-user support with hybrid authentication (email OTP as P1, OIDC social login as P3), data isolation, and session management. Users in public mode get isolated storage directories while shared system resources remain read-only. Technical approach uses JWT tokens, Mailgun for email OTP delivery, file-based persistence for user data and temporary auth state, and service-layer permission enforcement.

## Technical Context

**Language/Version**: Python 3.11+ (backend), JavaScript ES6+ (frontend with Alpine.js 3.x)
**Primary Dependencies**: FastAPI (backend API framework), Alpine.js 3.x (frontend reactivity), Tailwind CSS 3.x (UI styling), authlib (OIDC client), PyJWT (JWT tokens), requests (Mailgun API), pydantic (validation)
**Storage**: File-based JSON storage (`/data/` directory structure with `/data/shared/`, `/data/users/{user-id}/`, `/data/auth/`)
**Testing**: Chrome DevTools MCP for frontend user flows, integration tests for Mailgun email delivery and OIDC flows
**Target Platform**: Linux containers (Podman/Docker), deployed via compose files, accessible via web browser
**Project Type**: Web application (backend + frontend separation)
**Performance Goals**: <500ms for resource list merging (shared + user), <1min OIDC auth flow, <2min email OTP delivery
**Constraints**: File-based storage only (no database), 24-hour session tokens, 15-minute OTP expiration, 5 OTP requests/hour rate limit
**Scale/Scope**: Personal project with optional public deployment, file-based auth state with in-memory caching, supports hundreds of users (not thousands)

Frontend adds login view and auth client logic. Data directory expanded with `/data/users/` and `/data/auth/` while preserving existing `/data/shared/` structure for backward compatibility.

### Permissions
- Self-hosted mode: All resources in `/data/shared/` (full CRUD)
- Public mode: `/data/shared/` (read-only system resources created by copying files pre-deployment), `/data/users/{user-id}/` (user's personal resources with full CRUD)
- No cross-user access permitted in public mode
- No admin users: All authenticated users have equal privileges
- Shared resource preparation: Developer manually moves JSON files to `/data/shared/` before public deployment
